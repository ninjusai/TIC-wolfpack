# Eval Spec: Wolf Pack Mission Control V1

## 1. Overview

This eval spec defines the measurable criteria for validating Wolf Pack Mission Control V1 — a persistent desktop interface that gives the Pack Owner visibility into project pipelines, artifact contents, database state, and structured project intake. Every eval case traces to a success criterion in [PRB-mission-control-001]. The system under test is a desktop application, not an AI model; therefore, the vast majority of scorers are **algorithmic** (deterministic, automated checks). No human-aligned AI scorers are used in V1.

Traces to: [PRB-mission-control-001]

## 3. Rubrics

### Rubric: RBR-mission-control-001 — Data Accuracy

| Field | Value |
|-------|-------|
| **ID** | `RBR-mission-control-001` |
| **Dimension** | Correctness of displayed data versus source of truth |
| **Scale** | Binary pass/fail per data point |
| **Criteria** | Each displayed value must exactly match the corresponding value in manifest.json, wolfpack.db, or artifact file |

#### Score Levels

| Score | Definition |
|-------|-----------|
| PASS | Every displayed data point matches the source of truth with zero discrepancies |
| FAIL | One or more displayed data points differ from the source of truth |

### Rubric: RBR-mission-control-002 — Completeness

| Field | Value |
|-------|-------|
| **ID** | `RBR-mission-control-002` |
| **Dimension** | Whether all expected data elements are present in the interface |
| **Scale** | Percentage (0-100) of expected elements displayed |
| **Criteria** | Count of elements displayed divided by count of elements that exist in the data source |

#### Score Levels

| Score | Definition |
|-------|-----------|
| 100% | All elements from the source are present in the interface |
| 80-99% | Most elements present; some are missing but the interface is functional |
| < 80% | Significant data elements are absent from the interface |

### Rubric: RBR-mission-control-003 — Performance

| Field | Value |
|-------|-------|
| **ID** | `RBR-mission-control-003` |
| **Dimension** | Responsiveness of UI interactions |
| **Scale** | Milliseconds |
| **Criteria** | Time from user action (click, navigation) to data fully rendered in the interface |

#### Score Levels

| Score | Definition |
|-------|-----------|
| PASS | Action completes and data renders in <= 2000ms |
| WARN | Action completes in 2001-3000ms |
| FAIL | Action takes > 3000ms or does not complete |

### Rubric: RBR-mission-control-004 — Intake Output Conformance

| Field | Value |
|-------|-------|
| **ID** | `RBR-mission-control-004` |
| **Dimension** | Whether intake form output conforms to the defined intake.json schema |
| **Scale** | Binary pass/fail |
| **Criteria** | Output file must be valid JSON, contain all six intake fields, and match the schema expected by Alpha |

#### Score Levels

| Score | Definition |
|-------|-----------|
| PASS | Output is valid JSON with all six fields present and correctly typed |
| FAIL | Output is invalid JSON, or any field is missing or incorrectly typed |

## 5. Scorers

### Scorer: SCR-mission-control-001 — Data Match Checker

| Field | Value |
|-------|-------|
| **ID** | `SCR-mission-control-001` |
| **Type** | `algorithmic` |
| **Applies to** | EVL-CASE-mission-control-003, 005, 006, 007, 015, 016, 017, 019, 020 |

- **Check:** For each displayed data point, extract the value from the UI (via DOM query or accessibility tree) and compare it to the corresponding value in the source file (manifest.json field, wolfpack.db query result, or artifact file content).
- **Pass condition:** All compared values are identical (string equality for text, numeric equality for numbers, boolean equality for flags). Zero mismatches.

### Scorer: SCR-mission-control-002 — Completeness Counter

| Field | Value |
|-------|-------|
| **ID** | `SCR-mission-control-002` |
| **Type** | `algorithmic` |
| **Applies to** | EVL-CASE-mission-control-003, 004, 005, 011, 014 |

- **Check:** Count the number of expected elements (projects from manifest files, pipeline stages, artifact files in directory, DB tables) and count the number of displayed elements in the UI. Compute: `displayed_count / expected_count * 100`.
- **Pass condition:** Result = 100%. All expected elements are present.

### Scorer: SCR-mission-control-003 — Performance Timer

| Field | Value |
|-------|-------|
| **ID** | `SCR-mission-control-003` |
| **Type** | `algorithmic` |
| **Applies to** | EVL-CASE-mission-control-018, 025, 026, 027 |

- **Check:** Measure elapsed time from trigger event (click, file change) to UI render-complete event (last DOM mutation in the target view area). Use performance.now() timestamps or equivalent instrumentation.
- **Pass condition:** Elapsed time <= the threshold defined for the specific eval case (2000ms for project switch, 5000ms for file watcher, 3000ms for manual reload).

### Scorer: SCR-mission-control-004 — File Output Validator

| Field | Value |
|-------|-------|
| **ID** | `SCR-mission-control-004` |
| **Type** | `algorithmic` |
| **Applies to** | EVL-CASE-mission-control-008, 009, 010 |

- **Check:** After intake form submission: (1) verify `artifacts/{project}/intake.json` exists, (2) parse as JSON (must not throw), (3) validate all six fields are present as keys, (4) validate field types match schema (problem: string, users: string[], scope_in: string[], scope_out: string[], constraints: string[], success_criteria: string[], prior_art: string[]), (5) compare field values to what was entered in the form.
- **Pass condition:** All five checks pass. For EVL-CASE-010 (validation failure), the inverse: file must NOT exist and UI must display a validation error element.

### Scorer: SCR-mission-control-005 — Session Persistence Checker

| Field | Value |
|-------|-------|
| **ID** | `SCR-mission-control-005` |
| **Type** | `algorithmic` |
| **Applies to** | EVL-CASE-mission-control-001, 002 |
